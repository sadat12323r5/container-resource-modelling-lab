Capacity Planning Concepts
Capacity planning determines resources to maintain performance under 
varying demand. Data on workload and latency from real life data, used 
to simulate the system based on workload and capacity.
Integration of Traditional and Modern Methods
Combines analytical models with monitoring data and machine learning 
for predictive planning.

Industry shift toward machine learning–assisted capacity planning to 
address scalability and dynamic workloads in cloud systems: 
-> Vitui & Chen (2021) introduced MLASP, which leverages historical workload traces and performance 
counters to train models predicting resource bottlenecks under varying CPU and memory allocations 
— demonstrating improved forecasting accuracy compared to static models.
 
-> Müller et al. (2022) applied supervised regression and anomaly detection to automate capacity 
adjustments in enterprise IT systems, replacing manual threshold tuning.
 These ML-driven approaches mirror real-world practices at large-scale providers (e.g., AWS, Google 
Cloud) and suggest pedagogical value.
->Marin (2023) demonstrated that teaching performance evaluation through operational laws (U = XS, 
N = XR) and DES frameworks significantly improves conceptual understanding compared to purely 
theoretical instruction.
My take-away was students could replicate simplified ML pipelines on containerized platforms to 
visualize prediction-driven resource scaling, or also do DES with real world data collected from docker 
logs. The workload traces from the Docker logs can serve as the data the students work with.

Monitoring Tools for modelling:
Modern capacity analysis increasingly depends on a composite toolchain combining real-time 
monitoring, metrics visualization, and load testing.
->  Zhang et al. (2007) introduced R-Capriccio, an integrated tool uniting workload analysis, anomaly 
detection, and forecasting — an early blueprint for modern systems like Prometheus and Grafana.
-> In educational and experimental contexts, DocLite (Varghese et al., 2016) demonstrated lightweight 
benchmarking for cloud containers, automating the extraction of metrics like CPU usage, I/O latency, and 
throughput from Docker APIs.  Pairing cAdvisor (live container metrics), Prometheus (data storage and 
queries), Grafana (visualization), and Vegeta (HTTP load generation) offers a cost-efficient, open-source 
alternative with minimal setup overhead — ideal for academic capacity planning experiments.
-> Comparative research (Casalicchio & Perciballi, 2017; Arango et al., 2017) also confirmed that Docker’s 
overhead is low enough for teaching performance modeling without distorting key metrics.
Conclusion:  So, I decided to use cadvisor for for showcasing live container metrics for my docker 
applications. Then prometheus and grafana can be used later for doing more data visualisation and 
analysis.

What cAdvisor does:
cAdvisor is like a real-time probe living inside Docker. It
• Monitors container resource usage (CPU, memory, I/O, network).
• Exposes those metrics live at http://localhost:8081/metrics in both web UI 
and Prometheus format.
• Keeps a few minutes of data in memory (not meant for long-term storage).
• Can’t easily do comparisons, alerts, or queries over time. (which is where 
prometheus comes in. Then grafana has a more friendly UI for displaying 
those very data.)
What Prometheus + Grafana does:
• Capture historical data for DES parameter fitting.
• Analyze steady-state vs transient behavior.
• Reuse datasets for ML-based capacity planning.
• Provide teaching dashboards for students to interact with their own system 
metrics.

Docker CPU quota = 1 core Service capacity limited to one server

Request completion time (from logs) Service time distribution ( S )
cAdvisor / Prometheus metrics Data source for estimating ( λ,μ,U,R)

We're likely modelling a single-server queue with Poisson arrivals(Vegeta open loop) and 
general service times:
So, basically it's a M/G/1 queue.
• M → “Memoryless” arrivals → exponential inter-arrival times (thanks to open-loop Vegeta)
• G → “General” service time distribution → because actual request processing time isn’t 
perfectly exponential
• 1 → one server/core/thread handling requests sequentially
Here, my λ is the vegeta rate, S (service time distribution) is from the server-side response time 
logs or cAdvisor CPU time per request. And I derive the service rate from the mean service 
time, μ=1/E[S]. Then I can find my utilisation by comparing to cAdvisor total CPU usage, and 
finally the Response times from my Vegeta latency reports.

Threats to Validity and accuracy
Docker Desktop introduces variability due to host OS scheduling and 
virtualization overhead.
Network stack jitter and Prometheus scrape intervals limit temporal 
resolution.
CPU throttling granularity under cgroups can produce timing
artifacts near saturation points
To mitigate these risks, experiments were conducted under minimal 
background load, CPU cores were pinned using --cpuset-cpus, and repeated 
runs were averaged with confidence intervals calculated. While these 
mitigations reduce variance, they do not eliminate all noise; thus, results are 
interpreted as representative rather than absolute.
