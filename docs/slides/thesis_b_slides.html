Motivation

The original assumption was simple: service time is independent of load

response time = queue time + service time rho = lambda * E[S] / c

Classical M/G/c assumes the distribution of S is fixed.

Capacity planning question

Given a measured trace of a containerised service, can we predict the response-time distribution at higher load without running the service at that load?

Early Discovery

Instrumentation had to be fixed before the model could be trusted

  • Initial service-time logging confused service demand with handler wall-clock duration.
  • Apache initially served the wrong route but still returned HTTP 200.
  • Docker CPU limits needed both quota and CPU pinning.
  • Concurrent CSV writes required locking.
  • Rejected requests had to be excluded near saturation.
The difficulties were not side stories. They explain why the final traces are credible.

Pivot6
Core Finding

For some CPU-bound servers, service time grows with load

The M/G/c constant-service-time assumption is not always valid for real containerised services.

Wall-clock service time can include scheduler delay when request handlers compete for pinned CPU cores.

Core finding7
Mechanism

OS/runtime time-sharing explains the service-time inflation

  • Multiple handlers become runnable under load.
  • The scheduler time-slices the pinned CPU core.
  • Each handler performs similar CPU work, but progresses more slowly in wall-clock time.
  • This gives Processor-Sharing-like behaviour without claiming the application is literally M/M/1-PS.
E[S](rho0) = S0 / (1 - beta*rho0) rho0 = lambda*S0/c
Mechanism8
Models

M0, M1, and M2 are three service-time hypotheses

ModelFormulaMeaning
M0E[S] = S0Classical constant service time
M1E[S] = S0 + alpha*rho0Linear empirical growth
M2E[S] = S0 / (1 - beta*rho0)Hyperbolic PS-motivated growth

The thesis claim is not that M2 always wins. The claim is that M0 fails in important workloads, and M2 gives a mechanistic correction.

Does the better service-time model improve response-time simulation?

  • Fit quality alone is not enough.
  • The DES must compare response-time CDFs.
  • The key comparison is CDF_REF vs CDF_M0, CDF_M1, and CDF_M2.
CDF_REF = measured response times CDF_M0 = DES with constant S0 CDF_M1 = DES with linear E[S] CDF_M2 = DES with hyperbolic E[S]
Validation logic11
CDF Evidence

M2 clearly beats M0 in selected failure cases

  • Python DSP 3c at 200 rps: KS M0 0.2943 -> KS M2 0.1153.
  • Apache DSP 1c at 75 rps: KS M0 0.4828 -> KS M2 0.3671.
  • Go lognormal 1c at 200 rps: KS M0 0.1815 -> KS M2 0.0925.

This is the strongest innovation case: M0 fails, so a load-dependent model is needed.

CDF results12
Full Comparison

M1 is often competitive, which makes the final claim more careful

  • M2 is not best everywhere.
  • M1 can approximate finite-range service-time growth.
  • M2 matters because it is mechanistic and has saturation behaviour.

Why this matters

  • The programmed work distribution was fixed.
  • Measured wall-clock service time still increased under load.
  • The likely cause is goroutine/runtime scheduling inside the measured interval.

Structural Findings

Not every mismatch is solved by changing service time

Node.js cluster

Not a shared M/G/3 queue. It is closer to 3 independent M/G/1 queues with splitting.

Go SQLite

Application queue feeds a hidden SQLite/WAL lock queue: a tandem system.

Java DSP

JIT warmup changes service time for runtime reasons, not queueing reasons.

Thesis C19
Closing

The revised thesis statement

Standard M/G/c works only when service time is exogenous. In real containerised runtimes, service time can itself become load-dependent.

The value of Thesis B is identifying when this happens, why it happens, and how to correct the DES when M0 fails.
