Motivation

The starting problem was practical capacity planning

response time = queue time + service time rho = lambda * E[S] / c

The first goal was to predict response-time distributions from measured traces.

Capacity planning question

Given a measured trace of a containerised service, can we predict what happens as load increases, and can we tell which modelling assumption breaks when prediction fails?

Thesis Framing

The project has two connected goals

1. Teaching artefact

A runnable Docker lab where students can observe queueing behaviour, saturation, response-time CDFs, service-time distributions, and DES validation on their own machine.

2. Capacity-planning test-bed

Modern capacity planning is mostly reactive, not explanatory

Dashboards show symptoms

Prometheus/Grafana can show CPU and latency rising, but not which queueing assumption broke.

Autoscalers react late

Design Rationale

The server catalogue was designed around modelling contrasts

Same workload, different runtimes

DSP-AES was repeated across languages so differences could be attributed to runtime/concurrency, not application logic.

CPU-bound vs I/O-bound

CPU-bound DSP exposes processor scheduling; SQLite exposes database locks and I/O waiting.

Single-core controls

One pinned CPU core makes saturation and scheduling effects easier to observe.

Multi-worker variants

Three-worker versions test whether M/G/c with shared workers is the right abstraction.

Expected control cases

Python 1c and some low-load cases should behave closer to constant service time.

Expected mismatch cases

Node cluster and SQLite were likely to expose architectures beyond a single shared queue.

Server design5
Workload Choice

DSP-AES was chosen to make CPU work comparable across languages

  • Each DSP request performs AES decrypt, FIR filter, then AES encrypt.
  • The input size is fixed, so the intended CPU work is similar from request to request.
  • This makes service-time changes with load more meaningful: the workload did not intentionally become harder.
  • The same DSP task across Apache, Node, Python, and Java highlights runtime behaviour.
  • Go was included separately through a synthetic lognormal server and SQLite-backed server.
fixed request size fixed computation path different runtime architecture = comparable capacity experiment

Server rationale5
Runtime Terms

Some server behaviours need runtime vocabulary

GIL

Python's Global Interpreter Lock allows only one thread to execute Python bytecode at a time. In the 1-worker case, this suppresses CPU competition.

JIT

Just-In-Time compilation lets the JVM compile hot code paths while the experiment runs, so early requests can be slower than later ones.

WAL

SQLite Write-Ahead Logging improves concurrency, but write/lock paths can still serialise internally and create a hidden queue.

Tandem queue

A request passes through two queues in sequence, e.g. application worker first, database lock second. A single M/G/c model cannot represent both stages.

Docker Configuration Work

The Docker setup had to enforce actual experimental control

CPU quota plus pinning

cpus limits share; cpuset pins cores. Both were needed so "1-core" actually meant one physical core.

One service per load run

Only one target server should run during measurement to avoid cross-container CPU contention.

Bind-mounted logs

Containers write per-request CSV traces into data/live_logs/ for later analysis.

Correct Apache mounts

The Apache config must be mounted as a file, not accidentally created as a directory. This caused an earlier Docker startup error.

Pipeline7
Data Collection

Each load run produces a request-level trace

  • The load generator sends requests at a fixed average rate using exponential inter-arrival times.
  • The server records service time inside the handler around the actual work.
  • The client/request path records response time.
  • Rejected or non-2xx requests are filtered for stable-model validation.
arrival_unix_ns service_ms queue_ms response_ms status_code
Trace format8
Trace Interpretation

The trace separates three timing quantities

|--- queue_ms ---|--- service_ms ---| |----------- response_ms -----------| response_ms = queue_ms + service_ms
  • service_ms: measured server work interval.
  • queue_ms: inferred waiting before service.
  • response_ms: client-observed latency used for CDF validation.
  • The early mistake was measuring too much handler wall-clock time as service.
Timing data9
DES Mechanics

The DES replays arrivals through a queueing model

  • Read the observed arrival sequence from the trace.
  • Keep a list/heap of worker availability times.
  • For each request, assign it to the next available worker.
  • If the worker is busy, the difference becomes simulated queue time.
  • Sample or replay service time, then record simulated response time.
start_i = max(arrival_i, worker_free_at) finish_i = start_i + service_i resp_i = finish_i - arrival_i

Validation Metric

CDF overlays and KS distance decide whether the model works

  • The measured response-time CDF is the reference.
  • Each DES run produces a simulated response-time CDF.
  • KS distance is the maximum vertical gap between the two CDFs.
  • Large KS means either the service distribution or the queueing structure is wrong.
KS = max_x | F_ref(x) - F_sim(x) |

Research Journey5
Baseline First

The first modelling step was standard M/G/1 and M/G/c validation

Assume fixed service distribution

The standard model treats service times as independent of arrival rate.

Use observed arrivals

Replay mode removes arrival-process error by using the actual trace arrival sequence.

Compare response CDFs

If the CDFs match, the queueing abstraction is probably adequate.

Inspect high KS cases

If the CDFs do not match, the failure points to a hidden queue, wrong worker structure, or changing service time.

Only then fit load models

The load-dependent service model was introduced after baseline mismatch appeared in real traces.

No shortcut

The discovery came from debugging failed M/G/c predictions, not assuming load dependence at the start.

Original ML Plan

The plan was to learn service-time distributions from Docker traces

  • Use KDE/GMM/regressors to learn empirical service-time distributions.
  • Feed learned distributions into DES.
  • Use ARIMA/LSTM-style forecasting to predict future arrival rates.

Original ML plan9
Why The Plan Changed

The data did not need a black-box service-time learner

Shape was stable

Many service-time distributions remained roughly lognormal/right-skewed.

Mean moved with load

The dominant effect was not arbitrary distribution shape; it was mean inflation.

One variable mattered

Nominal utilisation rho0 explained much of the change.

Mechanism existed

OS/runtime scheduling gave a physical explanation for wall-clock inflation.

Less ML, more insight

A two-parameter model was more interpretable than KDE/GMM black-box fitting.

ML still has a role

Forecasting future arrival rate remains an ML/time-series problem.

Pivot reason10
Revised Architecture

The useful hybrid system shifted from service-time ML to proactive forecasting

Prometheus / traces -> forecast lambda(t + delta) -> load-dependent DES using S0, beta/theta, CV, c -> predicted response CDF / SLO risk -> scale before the queue builds

This is the industrial scaling story: feedforward autoscaling instead of reactive threshold scaling.

Revised architecture11
Core Finding

For some CPU-bound servers, service time grows with load

The M/G/c constant-service-time assumption is not always valid for real containerised services.

Wall-clock service time can include scheduler delay when request handlers compete for pinned CPU cores.

Core finding7
Mechanism

OS/runtime time-sharing explains the service-time inflation

  • Multiple handlers become runnable under load.
  • The scheduler time-slices the pinned CPU core.
  • Each handler performs similar CPU work, but progresses more slowly in wall-clock time.
  • This gives Processor-Sharing-like behaviour without claiming the application is literally M/M/1-PS.
Old M2: E[S](rho0) = S0 / (1 - beta*rho0) Exploratory M3 candidate: E[S](rho0) = S0 * (1 + theta*rho0/(1-rho0)) rho0 = lambda*S0/c
Mechanism8
Models

The model family now separates fit from interpretation

ModelFormulaMeaning
M0E[S] = S0Classical constant service time
M1E[S] = S0 + alpha*rho0Linear empirical growth
M2E[S] = S0 / (1 - beta*rho0)Original hyperbolic PS-motivated fit
M3E[S] = S0*(1 + theta*rho0/(1-rho0))Base work plus CPU-contention penalty

M3 avoids claiming there is literally a Processor-Sharing queue inside FCFS. It is a reduced-form contention correction.

Model Stress Test

The contention-penalty model stands, but does not dominate

  • Service-time fit: M3 is almost tied with M2 for the strong 1-core CPU cases.
  • DES response CDFs: average KS was M2=0.4940, M3=0.4955.
  • M3 wins some cases, e.g. Python DSP 3c at 200 rps.
  • Best conclusion: M3 is a more realistic interpretation, while M2 remains marginally stronger overall by KS.
M2 interpretation: PS-shaped fitted slowdown M3 interpretation: base service work + scheduler contention penalty
Alternative equation10
Priority Models Tried

Explicit priority structures were tested as alternatives

  • Preemptive-resume M/G/1: DSP jobs are low priority; synthetic short high-priority jobs can interrupt and DSP resumes later.
  • Feedback priority: each DSP job receives a high-priority first quantum, then unfinished work returns to a lower-priority queue.
  • These models improved over plain FCFS_M0 in some traces.
  • They did not beat M2 overall: average KS was M2=0.3727, preempt=0.4054, feedback=0.4215.

Structural Findings

How the non-service-time mismatches were diagnosed

Node.js cluster

Observed: M/G/3 DES stayed high-KS even after using c=3 and fitted service times.
Reason: Node cluster routes requests into separate worker event loops, so arrivals are split across 3 M/G/1 queues rather than sharing one queue.

Go SQLite

Observed: Service mean was not enough to explain heavy tails and irregular CDF mismatch.
Reason: requests wait first for the Go worker, then again inside SQLite/WAL locks, so the path is a two-stage tandem queue.

Java DSP

Observed: service time decreased after early requests instead of increasing with load.
Reason: JIT compiles hot code paths during warmup, so later requests run faster. That is a runtime phase change, not queue growth.

Inference18
Tandem Queue Thinking

Go SQLite suggests two queues in sequence, not one bigger queue

request -> application worker queue -> SQLite / WAL lock queue -> response
  • A single M/G/c model only sees one waiting room and one service stage.
  • SQLite can add another internal waiting point through locks and disk/cache behaviour.
  • End-to-end response time is shaped by both stages.
  • This is why parameter tuning alone cannot fix Go SQLite.

Closing

The revised thesis statement

Standard M/G/c works only when service time is exogenous. In real containerised runtimes, service time can itself become load-dependent.

The value of Thesis B is identifying when this happens, why it happens, and how to correct the DES when M0 fails.
